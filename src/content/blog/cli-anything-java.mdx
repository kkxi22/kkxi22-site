## CLI-Anything 是什么?

CLI-Anything 的核心作用，是**把原本只有 GUI 或现成代码库的软件，自动生成为 AI agent 可调用的 CLI 接口**。

它主打的不是"再造一个终端工具"，而是**让任意软件变成 agent-native**：AI 不再靠点屏幕、找坐标、做脆弱的 GUI 自动化，而是通过结构化命令直接操作软件。

> 你有一个现成软件或项目，比如 Blender、GIMP、内部业务系统、桌面应用，原本 AI 很难稳定操作；CLI-Anything 试图用"一条命令"为它生成一层 CLI harness，让 Claude Code、Codex、Cursor 这类 agent 可以像调用命令行工具一样调用它。

官方描述里强调了几点：CLI 对 agent 来说天然友好，因为它是**文本化、可组合、可通过 `--help` 自描述、还能输出 JSON**，便于 AI 可靠解析和串联工作流。

## CLI-Anything 的实际价值

### 1. 把软件能力"结构化"暴露给 AI

原来是"打开页面→点按钮→填表单"，现在变成类似 `app order create --sku xxx --count 2` 这种可预测命令。这样 agent 更稳。

### 2. 替代一部分脆弱的 GUI 自动化

相比纯视觉识别、坐标点击、DOM 猜测，这种 CLI 包装通常更稳定、更可测试。

### 3. 降低给 AI 接入旧系统的成本

不是每个系统都先重写一套 MCP server 或正式 API；CLI-Anything 的卖点就是先自动生成一层 agent 能消费的命令接口。

### 4. 让软件更容易接入 Claude Code / Codex / OpenClaw 一类 agent 工作流

官方站点和 GitHub 都把它定位成给这些 agent 生态使用的桥接层。

## 重要注意

<Callout>
  <strong>注意：</strong> CLI-Anything 不是"AI 自动控制任何软件的魔法按钮"。它更像一个 CLI 生成器 / agent 适配层。
</Callout>

官方站点明确写了，它本身不是独立终端工具，而是一个会为特定软件生成完整 CLI harness 的插件式方案。也就是说，它更偏"把软件工程化包装成 agent 能稳定调用的接口"，不是简单录制点击脚本。

**一句话总结：CLI-Anything 的作用，就是把任意软件的能力转成 AI 更容易理解和调用的命令行接口，从而让软件变成 agent-native。**

## Java 微服务电商 + 乾坤微前端如何 Agent 化转化

### 一、先判断你这个电商系统里，哪些东西该"转"，哪些不该"转"

一个典型电商前后端分离系统，大致有这些域：

- 用户与认证：登录、注册、会员、地址
- 商品域：SPU/SKU、库存、价格、促销
- 交易域：购物车、下单、优惠、支付、退款
- 履约域：发货、物流、签收、售后
- 运营域：商品管理、活动配置、订单审核、客服后台
- 前端壳层：门户、店铺页、商品详情、购物车页、结算页

这里面最适合 agent 化的是：

#### 第一类：后端可结构化的业务能力

比如：
- `search_products`
- `get_product_detail`
- `add_to_cart`
- `create_order`
- `apply_coupon`
- `pay_order`
- `refund_order`
- `query_logistics`
- `update_inventory`
- `publish_product`
- `review_order_risk`

这些天然就应该变成 **API / Tool / MCP / CLI command**。

#### 第二类：后台运营能力

比如商品上架、活动配置、库存调整、价格批量修改、订单审核。这类经常本来就只有后台页面，没有好 API。这里 CLI-Anything 这类"把现有软件变成结构化命令"的思路就比较有价值，因为它的优势正是把现有代码库抽出可调用命令，而不是依赖 GUI 自动化。

#### 第三类：必须依赖真实前端交互的能力

比如：
- 用户登录态获取
- 风控验证码
- 第三方收银台支付
- 某些必须浏览器环境才能完成的 OAuth / WebAuthn
- 某些活动页、搭售页、营销弹层的真实前端逻辑验证

这些不要强行 CLI 化，而是保留 **browser agent / Playwright / bb-browser** 方式兜底。

### 二、最合理的转化目标，不是一个 CLI，而是"三层架构"

#### 第 1 层：领域微服务层

你现在已有 Java 微服务，这层不要动大架构，只做两件事：

1. 补齐领域 API 的稳定契约
2. 给 agent 场景加"更粗粒度"的业务接口

例如你现在可能有很多细碎服务：
- product-service
- inventory-service
- promotion-service
- order-service
- payment-service
- user-service

对人类前端来说，细碎 API 没问题；但对 agent 来说，应该再提供一层 **Agent Facade / BFF for Agent**：

- `agent-product-service`
- `agent-order-service`
- `agent-ops-service`

比如原来下单要 8 个接口串起来，agent 最好只调一个：

```javascript
create_order_from_cart(userId, cartId, couponCode, addressId, paymentMethod)
```

而不是让 agent 自己推理 8 步。

#### 第 2 层：Agent 接入层

这层才是"转化核心"。你可以同时暴露三种形式：

- **HTTP Tool API**：给 OpenClaw、内部 Agent 平台、前端 Copilot 用
- **MCP Server**：给 Claude/Codex/OpenAI 工具生态用
- **CLI**：给 CLI agent、终端自动化和本地批处理用

这里 CLI-Anything 能发挥作用的地方是：如果某些内部工具已经有独立代码库，或者某些后台能力散落在项目里，它可以帮助快速生成 CLI harness，让 agent 先用起来。它强调 CLI 对 agent 的优势包括结构化、可组合、自描述和 JSON 输出。

#### 第 3 层：浏览器执行层

只处理这几类事情：

- 登录
- 验证码
- 需要真实浏览器会话的用户操作
- 没有 API 的遗留页面操作
- 前端渲染验证与冒烟测试

这层可以是：
- Playwright
- browser-use / bb-browser 类方案
- 你自建 browser agent

### 三、你的"乾坤微前端"应该怎么转

乾坤本质是**前端壳 + 多个子应用**。从 agent 化角度看，乾坤不是阻碍，反而给了你一个很好的拆分边界。

#### 1. 不要把整个乾坤应用当成一个整体转

而是把每个子应用识别成一个能力域：

- `mfe-product`
- `mfe-cart`
- `mfe-checkout`
- `mfe-order`
- `mfe-customer-service`
- `mfe-admin-product`
- `mfe-admin-order`

然后分别判断：
- 这个子应用的核心能力，是否已有后端 API？
- 是否只是前端壳拼装？
- 是否有大量纯前端状态机逻辑？
- 是否依赖浏览器上下文？

#### 2. 对用户侧子应用：优先提炼"意图级 API"

例如：

**商品详情页**
- 页面行为：切换 SKU、查库存、看促销、加入购物车
- agent 能力：`get_sku_options`, `check_stock`, `preview_price`, `add_to_cart`

**结算页**
- 页面行为：地址选择、优惠券试算、运费估算、提交订单
- agent 能力：`estimate_order`, `recommend_best_coupon`, `submit_order`

也就是说，**把前端里的业务编排抽回后端 Agent Facade**。

#### 3. 对运营后台子应用：可以考虑 CLI-Anything 辅助生成 CLI

尤其是：
- 批量商品上下架
- 导入导出
- 促销活动配置
- 订单人工审核
- 售后单处理

这些很多系统里都沉淀在后台页面代码和内部接口里。CLI-Anything 的思路适合用来快速把现有后台工具"工具化"，但建议优先用于**内控后台**，不要直接上核心交易主链路。

### 四、推荐你采用的实际转化路线

#### 路线 A：最稳妥，适合正式生产

**后端 API First + MCP/CLI 包装 + 浏览器兜底**

##### 步骤 1：盘点电商核心意图

先列用户真正会对 agent 说的话：

- 帮我找 300 元以内的无线耳机
- 帮我比较这 3 个 SKU
- 用最优优惠券下单
- 查一下订单物流
- 把这个商品批量调价 5%
- 帮我处理未发货超时订单

这些不是页面动作，而是**业务意图**。

##### 步骤 2：把意图映射为领域工具

比如：
- `product.search`
- `product.compare`
- `cart.add`
- `order.preview`
- `order.submit`
- `payment.create`
- `payment.pay`
- `logistics.track`
- `ops.product.bulk_update_price`

##### 步骤 3：每个工具绑定后端聚合 API

由 Java 侧提供聚合接口，而不是让 agent 直连几十个微服务。

##### 步骤 4：自动生成三种出口

- REST/OpenAPI
- MCP tools
- CLI commands

CLI 命令例子可以长这样：

```bash
shop product search --keyword "蓝牙耳机" --max-price 300 --json
shop order preview --cart-id xxx --coupon AUTO --address-id xxx --json
shop order submit --cart-id xxx --payment-method wechatpay --json
shop ops product bulk-price-adjust --category "耳机" --percent -5 --dry-run --json
```

##### 步骤 5：仅在必要处接浏览器 agent

比如：
- 首次登录
- 支付确认
- 图形验证码
- 站外 OAuth
- 某些埋在前端页面里的遗留逻辑

这是最推荐的路线。

适合这几类场景：
- 你暂时没有精力改后端聚合层
- 想先验证 agent 是否真能创造价值

先做：

这类页面的风险比"用户真实支付下单"低得多，更适合先试。

### 五、Java 微服务侧应该补什么

你后端是 Java 微服务，我建议新增一个 **Agent Gateway / Agent Orchestrator**，而不是让 agent 直接调每个微服务。

#### 这个层要做的事

##### 1. 统一鉴权

区分三类身份：
- 用户代理执行：代表某个登录用户操作
- 员工代理执行：代表运营/客服/管理员操作
- 系统代理执行：批处理/巡检/推荐

每次调用都要带：
- actorId
- actorType
- tenantId / shopId
- session scope
- traceId
- approval policy

##### 2. 权限收敛

例如 agent 可以：
- 查商品
- 预估订单
- 创建待支付订单

但不能默认：
- 直接支付
- 批量退款
- 改库存
- 改价格

这些要加：
- 二次确认
- 风控审批
- 金额阈值
- 人工授权

##### 3. 幂等与补偿

电商核心链路一定要加：
- idempotency key
- saga / 补偿
- 乐观锁
- 操作日志

否则 agent 重试一下就可能多下单、多退款。

##### 4. 给 agent 友好的响应格式

不要只返回前端 DTO。最好返回：

```json
{
  "success": true,
  "status": "PENDING_PAYMENT",
  "message": "订单已创建，等待支付",
  "next_actions": [
    {
      "action": "pay_order",
      "requires_confirmation": true
    }
  ],
  "data": {
    "orderId": "xxx",
    "amount": 29900,
    "currency": "CNY"
  }
}
```

这样 agent 更容易串流程。

### 六、前端侧要补什么

虽然主战场在后端，但前端也要做一些改造，尤其你是乾坤微前端。

#### 1. 建立"页面能力清单"

对每个子应用标出来：
- 页面路由
- 关键用户任务
- 调用的 API
- 前端本地状态
- 哪些逻辑纯前端计算
- 哪些逻辑必须浏览器环境

这一步很重要，因为很多所谓"前后端分离"项目，真实业务逻辑其实散落在前端里。

#### 2. 把前端里的关键编排回收

常见要回收的逻辑：
- 优惠券最优匹配
- 运费与税费试算
- SKU 组合校验
- 营销活动命中规则
- 购物车合并策略
- 表单字段依赖校验

这些如果只存在前端，agent 很难稳定调用。

#### 3. 给浏览器 agent 预埋稳定锚点

给必须走页面的流程加：
- `data-testid`
- 语义化 action 名
- 结构化错误提示
- 页面状态 API
- 关键动作事件埋点

这样 browser agent 不会被乾坤切换、样式变化、弹层重绘轻易搞崩。

### 八、你这个架构下，CLI-Anything 具体放哪最合适

我给你一个明确判断：

#### 最适合
- 内部运营后台
- 历史后台工具
- 有代码库但缺正式 API 的桌面/GUI 工具
- 某些前端工程辅助工具
- 某些 admin 微前端子应用

#### 一般适合
- 中后台商品管理
- 营销配置
- 内容管理
- 报表导出

#### 不适合直接当主方案
- 核心交易链路
- 支付链路
- 风控强依赖链路
- 用户登录鉴权主链路

原因很简单：CLI-Anything 强项是"把现有软件变成 agent 可调用 CLI"，而不是替代一套严格治理、强一致、强审计的电商交易中台。官方材料也更强调 CLI harness、结构化命令和自动化生成，而不是面向高风险金融级交易的完整治理框架。

## 一句话总结

**你的电商系统不是"把乾坤前端转成 CLI"，而是"把 Java 微服务的业务能力提炼成 agent API，再把乾坤前端降级为浏览器兜底层"；CLI-Anything 更适合加速改造后台工具和遗留软件，不适合作为核心交易链路的唯一转化方式。**
