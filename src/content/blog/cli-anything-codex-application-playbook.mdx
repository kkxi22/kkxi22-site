<h2 id="principles">先讲原理：为什么这套模式可行</h2>

CLI-Anything 的核心不是“让 AI 会点 GUI”，而是把 GUI 软件变成可编排接口层。  
它依赖五个工程原则：

1. **Backend-first**：先操作原生数据结构，再调用真实后端渲染。
2. **真实软件硬依赖**：缺依赖就明确失败，不做低质量 fallback。
3. **双模式并存**：REPL 支持会话式交互，子命令支持脚本化批处理。
4. **`--json` 优先**：既能人类调试，也能给 Agent 稳定消费。
5. **产物验证优先**：不只看退出码，必须验证输出文件有效性和质量。

<Callout tone="success">
  <p>
    一句话：它做的是“真实软件能力接口化”，不是“重写一个替代软件”。
  </p>
</Callout>

<h2 id="architecture">再讲架构：Codex 在哪里发挥作用</h2>

从仓库结构看，职责分层很清晰：

- `cli-anything-plugin/`：方法论与命令模板（HARNESS 是权威 SOP）
- `opencode-commands/`：OpenCode 命令入口
- `codex-skill/`：Codex 技能入口（实验性、社区贡献）
- `<software>/agent-harness/`：每个软件的独立可安装 CLI 包

Codex skill 的作用是“接入层”，不是新架构。它复用了同一套 HARNESS 流程：

```text
自然语言任务 -> build/refine/validate 流程 -> 生成 cli-anything-<software> 包
            -> pip install -e . -> REPL/子命令调用 -> --json 给上层 Agent 编排
```

这就是为什么它能跨 Claude/OpenCode/Codex 保持一致产物结构。

<h2 id="how-to-use">如何使用：从安装到可运行</h2>

第一步，安装 Codex skill：

```bash
git clone https://github.com/HKUDS/CLI-Anything.git
bash CLI-Anything/codex-skill/scripts/install.sh
```

默认安装路径：

- `$CODEX_HOME/skills/cli-anything`
- 或 `~/.codex/skills/cli-anything`（未设置 `CODEX_HOME` 时）

第二步，在 Codex 中直接描述任务：

```text
Use CLI-Anything to build a harness for ./gimp
Use CLI-Anything to refine ./shotcut for picture-in-picture workflows
Use CLI-Anything to validate ./libreoffice
```

第三步，使用生成的 CLI：

```bash
cd <software>/agent-harness
pip install -e .

which cli-anything-<software>
cli-anything-<software> --help
cli-anything-<software>
cli-anything-<software> --json <command>
```

<h2 id="practical-recipes">应用层教程：五个可直接复用的落地模板</h2>

<h3 id="recipe-doc">模板 1：运营文档自动生成</h3>

1. 读取业务数据（订单、商品、渠道、周期）。
2. 生成/更新原生项目文件（如 ODF）。
3. 调真实 LibreOffice 导出 PDF/DOCX。
4. 校验关键字段和页数后归档。

这个模板适合先做试点，因为规则清晰、可验证性强。

<h3 id="recipe-ecom-visual">模板 2：电商素材批量出图</h3>

1. 固化一组模板参数（标题、价格、卖点、底图）。
2. 用 CLI 批量写入参数并调用 GIMP/Inkscape 导出。
3. 自动做分辨率/留白/文件大小检查。
4. 把不合格样本回流人工修订，再二次批跑。

这类任务的收益通常非常直接，能快速形成 ROI。

<h3 id="recipe-video">模板 3：短视频半自动剪辑</h3>

1. 基于素材清单构建时间线结构。
2. 自动插入片头片尾、字幕、音轨基础处理。
3. 交真实渲染后端输出视频。
4. 校验时长、码率和关键帧后进入审核。

关键点是把“重复操作”自动化，把“创意判断”留给人。

<h3 id="recipe-legacy">模板 4：企业旧 GUI 系统低侵入改造</h3>

1. 只抽最小闭环命令：`create/query/export/validate`。
2. 先从只读和低风险操作上线，积累日志与成功率。
3. 补齐 `--json` 输出和统一错误语义。
4. 再让上层 Agent 接审批、重试、通知等流程。

不重写系统也能进入 Agent 编排，这是这类方案的现实价值。

<h3 id="recipe-qa">模板 5：QA 回归自动验收</h3>

1. 对每次改动自动重建目标产物。
2. 检查格式结构、关键指标和业务规则。
3. 生成结构化差异报告（可直接投喂 Agent）。
4. 失败样本附带最小复现命令。

这能把“靠经验找问题”变成“靠证据定位问题”。

<h2 id="rollout">团队落地路线：四步走</h2>

1. 选一个高重复、低风险流程做首个试点。
2. 首批只做 5-10 条高频命令，不追求全覆盖。
3. 强制每条命令可 `--json`，并定义稳定错误语义。
4. CI 里跑真实软件 E2E 与安装后子进程测试，确保上线质量。

<h2 id="risks">边界与风险</h2>

1. **依赖风险**：真实软件未安装时必须清晰失败，不可静默降级。
2. **验证风险**：只测中间文件会漏掉渲染层问题，必须验最终产物。
3. **维护风险**：GUI 软件升级会影响接口映射，需要持续回归。
4. **范围风险**：如果一开始试图覆盖全部功能，项目很容易失控。

<h2 id="summary">总结</h2>

CLI-Anything × Codex 的价值不在“多一个技能入口”，而在把 GUI 软件 Agent 化变成可复制工程套路：  
**先后端、再命令、强验证、可发布、可编排。**

如果你要真正落地，建议从一个可量化业务流程试点，跑通后再扩展到跨团队场景。
