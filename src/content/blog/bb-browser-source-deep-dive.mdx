<h2 id="core-principles">核心原理</h2>

先说结论：`bb-browser` 的价值不在“自动化点击”，而在“把你的真实浏览器会话变成 Agent 可调用接口”。

它成立的关键有四个：

1. **真实会话优先**
   - 不重建登录态，不模拟新浏览器。
   - 直接用你已经登录的 Chrome 会话、Cookie、页面上下文。
2. **消息协议优先**
   - `ActionType` + `Request/Response` 是核心抽象。
   - CLI、daemon、extension 都围绕协议通信，降低耦合。
3. **语义引用优先**
   - 通过 snapshot/ref 把页面转成“先观察，再操作”。
   - 避免一上来就写脆弱 selector。
4. **站点适配器优先**
   - 高频站点逻辑沉到 `site` adapter。
   - 应用层能力可扩展，不把网站细节硬编码进核心引擎。

<h2 id="architecture">总体架构</h2>

从源码看，主链路非常清晰：

```text
AI Agent / CLI / MCP
        |
        v
    bb-browser CLI
        |
      HTTP
        v
      Daemon
        |
       SSE
        v
 Chrome Extension (MV3 Service Worker)
        |
        v
 chrome.debugger / scripting / tabs / storage
        |
        v
   User's Real Chrome Session
```

对应到代码职责：

- `packages/cli`: 参数解析、命令路由、MCP 入口 (`--mcp`)
- `packages/daemon`: `POST /command`、`GET /sse`、`POST /result`、`GET /status`
- `request-manager`: pending 请求和超时管理
- `sse-manager`: 与扩展长连接 + heartbeat
- `extension/background`: service worker、连接 daemon、处理命令
- `shared/protocol`: 行为类型和请求响应结构

为什么这样拆：

- daemon 专注“可靠转发和状态管理”
- extension 专注“贴近浏览器执行”
- CLI/MCP 专注“给人和 Agent 友好的调用入口”

<h2 id="quick-start">快速上手</h2>

这里给三个常见接入模式。

<h3 id="mode-openclaw">模式 A：OpenClaw 直连（最省事）</h3>

如果你已经在用 OpenClaw，这个模式最简单，不需要单独装扩展和 daemon：

```bash
bb-browser site update
bb-browser site reddit/hot --openclaw
bb-browser site xueqiu/hot-stock 5 --openclaw --jq '.items[] | {name, changePercent}'
```

适合：已经把 OpenClaw 作为浏览器执行层的人。

<h3 id="mode-standalone">模式 B：独立模式（Chrome Extension + daemon）</h3>

1. 安装 CLI

```bash
npm install -g bb-browser
```

2. 安装扩展
   - 从 Releases 下载扩展压缩包
   - 解压后在 `chrome://extensions/` 开启开发者模式并加载

3. 启动 daemon（默认 `127.0.0.1:19824`）

```bash
bb-browser daemon
```

4. 验证链路

```bash
bb-browser status
bb-browser open https://example.com
bb-browser snapshot -i
```

如果 `status` 显示扩展未连接，先检查扩展是否已加载、是否允许在当前浏览器会话运行。

<h3 id="mode-mcp">模式 C：MCP 接入 Claude Code / Cursor</h3>

在 MCP 配置里加：

```json
{
  "mcpServers": {
    "bb-browser": {
      "command": "npx",
      "args": ["-y", "bb-browser", "--mcp"]
    }
  }
}
```

然后在 Agent 侧走“snapshot -> ref -> click/fill”流程，不要直接 selector-first。

<h2 id="usage-recipes">应用层教程</h2>

这一节只讲可直接上手的任务模板。

<h3 id="recipe-research">教程 1：多平台研究（结构化输出）</h3>

```bash
bb-browser site arxiv/search "retrieval augmented generation" --json
bb-browser site twitter/search "RAG" --json
bb-browser site github/search "rag framework" --json
bb-browser site zhihu/search "RAG" --json
```

要做自动化汇总，建议每条命令都加 `--json`，再用 `--jq` 直接抽字段，减少后处理工作。

<h3 id="recipe-auth-fetch">教程 2：用登录态抓数据而不是自己补鉴权</h3>

```bash
bb-browser open https://example.com/dashboard
bb-browser fetch https://example.com/api/me --json
```

这套方式的核心优势是：不在脚本里手动拼 cookie/token，让页面会话自己完成身份上下文。

<h3 id="recipe-debug">教程 3：命令失败时的定位顺序</h3>

```bash
bb-browser network requests --with-body --json
bb-browser console --json
bb-browser errors --json
```

推荐排查顺序：

1. 先看 `errors`（页面脚本是否报错）
2. 再看 `network`（请求是否 401/403/429）
3. 再看 `console`（业务逻辑告警）

<h3 id="recipe-browser-ops">教程 4：通用浏览器操作编排</h3>

```bash
bb-browser open https://example.com
bb-browser snapshot -i
bb-browser click @3
bb-browser fill @5 "hello world"
bb-browser press Enter
bb-browser screenshot
```

实践建议：

- 每次页面变化后重新 `snapshot -i`
- ref 失效时不要硬点，先重新观察
- 尽量让 Agent 走“观察 -> 决策 -> 动作”闭环

<h2 id="adapter-workflow">如何把网站接入应用层</h2>

`bb-browser` 的长期扩展能力在 `site adapter` 体系，不在核心命令数量。

推荐工作流：

1. `bb-browser guide` 先看官方适配流程
2. 用 `network requests --with-body` 观察真实请求
3. 优先做“能稳定返回结构化 JSON”的最小版本
4. 通过 `site info` / `site list` 做命令面验证
5. 再补分页、过滤和错误处理

按 README 的经验，可把站点适配分为三层：

- Tier 1: 直接 cookie/fetch
- Tier 2: 需要 bearer/csrf
- Tier 3: 需要站内模块注入或更复杂上下文

<h2 id="limitations">边界与风险</h2>

使用前要明确三件事：

1. **信任边界**
   - 这是“代表你操作浏览器”的能力，不是隔离沙箱。
   - 只给可信 Agent 和可信脚本使用。
2. **扩展生命周期**
   - MV3 service worker 天生会休眠。
   - 项目已做 keepalive + 重连，但你仍要设计重试机制。
3. **平台策略变化**
   - 站点前端结构、风控策略会变。
   - adapter 层需要持续维护，不是一劳永逸。
