<h2 id="core-principles">核心原理</h2>

一句话结论：`bb-browser` 不是再造一个浏览器自动化框架，而是把“用户真实 Chrome 会话”转成 Agent 可编排执行面。

这套方案成立，依赖四个核心原则：

1. **真实会话优先**
   - 复用已登录浏览器的 Cookie、Session、页面 JS 运行态
   - 目标不是“模拟用户”，而是“代表用户当前会话执行”
2. **协议驱动优先**
   - 核心边界由 `ActionType / Request / Response` 统一定义
   - CLI、daemon、extension、MCP 共用同一套动作语义
3. **语义快照优先**
   - 先 `snapshot` 建立 `ref`，再 `click/fill/press`
   - 比 selector-first 更稳，抗 DOM 结构变化能力更高
4. **站点适配器优先**
   - 用 adapter 承载站点知识，主仓只负责控制平面和执行平面
   - 能力扩展靠“新增适配器”，而不是不停改内核命令

<h2 id="architecture">总体架构</h2>

两篇文档在架构判断上是一致的，主链路如下：

```text
AI Agent / CLI / MCP
        |
        v
    bb-browser CLI
        |
      HTTP
        v
      Daemon
        |
       SSE
        v
 Chrome Extension (MV3 Service Worker)
        |
        v
 chrome.debugger / scripting / tabs / storage
        |
        v
   User's Real Chrome Session
```

Monorepo 分层职责：

- `packages/shared`: 协议、类型、常量（跨层合同）
- `packages/cli`: 命令入口、参数解析、输出格式、site adapter 运行
- `packages/daemon`: 本地 HTTP + SSE 中转，做请求关联和超时治理
- `packages/extension`: MV3 扩展执行层，调用 CDP 真正操作标签页
- `packages/mcp`: 把浏览器能力封装成 MCP tools 给 Agent 调用

这类拆分的本质是“控制面/执行面解耦”：

- daemon 解决可靠通信，不碰浏览器细节
- extension 解决浏览器执行，不承载复杂交互入口
- CLI/MCP 解决人机调用，不直接管浏览器生命周期细节

<Callout tone="success">
  这也是 bb-browser 和传统脚本自动化的分水岭：它把“浏览器控制”做成了一套可复用协议系统，而不是零散脚本集合。
</Callout>

<h2 id="quick-start">快速上手</h2>

下面三种接法，覆盖大部分落地场景。

<h3 id="mode-openclaw">模式 A：OpenClaw 直连（最省事）</h3>

如果你已在用 OpenClaw，优先走这个模式：

```bash
bb-browser site update
bb-browser site reddit/hot --openclaw
bb-browser site xueqiu/hot-stock 5 --openclaw --jq '.items[] | {name, changePercent}'
```

适合：已有 OpenClaw 网关和浏览器能力的人，接入成本最低。

<h3 id="mode-standalone">模式 B：独立模式（Chrome Extension + daemon）</h3>

1. 安装 CLI：

```bash
npm install -g bb-browser
```

2. 安装扩展：
   - 从 Releases 下载扩展压缩包
   - 解压后在 `chrome://extensions/` 开启开发者模式并加载

3. 启动 daemon（默认 `127.0.0.1:19824`）：

```bash
bb-browser daemon
```

4. 验证链路：

```bash
bb-browser status
bb-browser open https://example.com
bb-browser snapshot -i
```

若 `status` 显示扩展未连接，先检查扩展加载状态、权限和 daemon 地址。

<h3 id="mode-mcp">模式 C：MCP 接入 Claude Code / Cursor</h3>

在 MCP 配置里加：

```json
{
  "mcpServers": {
    "bb-browser": {
      "command": "npx",
      "args": ["-y", "bb-browser", "--mcp"]
    }
  }
}
```

进入 Agent 执行时，优先走 `snapshot -> ref -> action` 闭环，不要直接 selector-first。

<h2 id="usage-recipes">应用层教程</h2>

这部分是两篇文档共同强调的重点：应用层比底层更决定价值。

<h3 id="recipe-research">教程 1：多平台研究（结构化输出）</h3>

```bash
bb-browser site arxiv/search "retrieval augmented generation" --json
bb-browser site twitter/search "RAG" --json
bb-browser site github/search "rag framework" --json
bb-browser site zhihu/search "RAG" --json
```

建议每条命令都输出 `--json`，再用 `--jq` 做最小字段抽取，便于汇总与二次处理。

<h3 id="recipe-auth-fetch">教程 2：用登录态抓数据而不是自己补鉴权</h3>

```bash
bb-browser open https://example.com/dashboard
bb-browser fetch https://example.com/api/me --json
```

重点不是“绕过鉴权”，而是“复用已存在的合法会话上下文”，减少自维护 token/cookie 的脆弱链路。

<h3 id="recipe-debug">教程 3：命令失败时的定位顺序</h3>

```bash
bb-browser network requests --with-body --json
bb-browser console --json
bb-browser errors --json
```

推荐排查顺序：

1. `errors`：前端运行是否直接异常
2. `network`：接口是否 401/403/429 或返回体异常
3. `console`：业务日志是否提示环境或参数问题

<h3 id="recipe-browser-ops">教程 4：通用浏览器操作编排</h3>

```bash
bb-browser open https://example.com
bb-browser snapshot -i
bb-browser click @3
bb-browser fill @5 "hello world"
bb-browser press Enter
bb-browser screenshot
```

实践建议：

- 每次页面变化后重新 `snapshot -i`
- ref 失效先重建快照，不要硬点旧引用
- 让 Agent 明确分三步：观察、决策、动作

<h2 id="adapter-workflow">如何把网站接入应用层</h2>

两篇文档都给出同一个结论：真正可规模化的是 adapter 体系，而不是堆命令。

推荐工作流：

1. 先用 `bb-browser guide` 明确适配边界
2. 通过 `network requests --with-body` 抓真实请求模型
3. 先做最小可用 adapter（稳定 JSON 输出）
4. 再补分页、过滤、重试和错误语义
5. 最后做命令面体验（`site info/list`、字段命名、默认排序）

可按复杂度分层推进：

- Tier 1: 直接 cookie/fetch
- Tier 2: 需要 bearer/csrf
- Tier 3: 需要站内模块注入或更复杂上下文

<h2 id="limitations">边界与风险</h2>

使用时要明确三类边界：

1. **权限与信任边界**
   - 这是用户会话级能力，不是沙箱能力
   - 只应授予可信 Agent 与可信脚本
2. **扩展生命周期边界**
   - MV3 service worker 可能休眠、断连、状态回收
   - 需要重连、重试和 ref 恢复策略
3. **站点变化边界**
   - DOM、接口、风控策略会持续变化
   - adapter 维护是常态，不是一次性开发

<h2 id="final-evaluation">最终评价</h2>

<Callout tone="success">
  <p><code>bb-browser</code> 的工程价值在于：把真实浏览器会话系统化为 Agent 能力，而不是只做脚本自动化。</p>
  <p>对实战最有帮助的路径是：先吃透“协议分层 + snapshot/ref 模型”，再把重心放在 adapter 和业务流程编排。</p>
</Callout>
